# AI Assistant Handbook (GEMINI.md)

このファイルは、本プロジェクトにおける **人間とAIの協業指針（Standard Operating Procedures）** を定めたものです。
AIを単なる「コード生成機」としてではなく、プロジェクトの規律（GEMINI.md）と開発プロセス（DEVELOPMENT.md）を遵守する **「バーチャル・チームメンバー」** として定義しています。

---

## 🛠 1. 開発の核心原則 (Core Principles)

### 🚨 エラーハンドリングの徹底 (Global Error Guard)
Windows実行ファイル（EXE）化した際、エラー発生時にウィンドウが即座に閉じてしまうのを防ぎ、サポート性を高めます。
- エントリーポイント（実行スクリプト）の最外周は、必ず `try...except` で囲むこと。
- 致命的なエラーをキャッチした際は、`traceback` を表示し、`input("エンターキーを押して終了してください...")` 等でユーザーの操作を待ち、ウィンドウを維持すること。

### 🇯🇵 ドキュメント・ファースト
本ツールは非エンジニアが利用することを想定し、「言葉の壁」を取り除きます。
- **コード内コメント**: 処理の意図（Why）が伝わるよう、丁寧な日本語で詳述すること。
- **ログ・メッセージ**: 状況が直感的にわかるよう、ユーザーフレンドリーな日本語で出力すること。

### 🛡 品質担保 (Quality Guard)
AI は常に「バグを未然に防ぐ」ベテランの視点を持ちます。
- **静的解析の重視**: エディタやツールが表示する Lint エラー（赤い波線など）を無視せず、可能な限り修正案を提示すること。
- **型ヒントの活用**: Python において、関数の引数や戻り値に型ヒントを積極的に追加し、コードの意図を明示化すること。

## 📦 2. 実行ファイル（EXE）互換性・配布ルール

PyInstallerによるビルドを成功させ、配布後のパス問題を回避するための「掟」です。

### 🏗 絶対インポートの徹底
- **ルール**: ルート起点の絶対インポート（例: `from common_utils.desknets.login import ...`）を必須とする。
- **理由**: EXE化された際のモジュール探索（Import Chain）の失敗を防ぐため。

### 📂 パス解決の標準化
設定ファイルやドライバ等のリソース参照時は、以下のハイブリッド方式を採用すること。
- `sys._MEIPASS` (EXE実行時) と `os.path.dirname(__file__)` (ソース実行時) の両方を考慮。

### 🔐 .env の外部化と配置
- `load_dotenv` は、実行ファイルの親ディレクトリ（`..`）も探索対象に含めること。
- 配布イメージ：`アプリ本体/app.exe` ➔ `その上の階層/.env`

## 📝 3. コミュニケーションと記録 (Work Journaling)

対話の透明性と、意思決定の履歴を保護するためのルールです。

### 📓 ワークジャーナル (work_journal/)
- **役割**: なぜその設計（A案ではなくB案）を選んだのかのバックストーリーを記録する。
- **タイミング**: 新機能追加時や、大きな設計変更の前後に必ず作成・更新すること。

### 📄 仕様書（SPECIFICATION.md）の同期
- 開発が進むにつれ、常にルートおよび各フォルダ의 `SPECIFICATION.md` を最新に保つこと。

## 🏗 4. 規模別・開発アプローチ (Scalability Strategy)

将来のチーム開発を意識し、意図的にフェーズを切り替えて開発を行います。

### 🟢 段階 1：個人開発 (Solo)
- **目的**: 最小限のドキュメントで高速にプロトタイプを構築。

### 🟡 段階 2：チーム開発 (Team)
- **目的**: 設計書の整備、コードレビューを意識した「他人が読めるドキュメント」の維持。

### 🔴 段階 3：大規模開発 (Enterprise)
- **目的**: 疎結合なモジュール設計、CI/CDを見据えた自動テストの強化。

## 🛡 5. Git 運用と文脈保護 (Context Protection)

無駄な「手戻り」を防ぎ、開発の履歴を美しく保つための制約です。

### ⚓️ チェックポイントの義務化 (Atomic Commits)
- **ルール**: 大きな作業フェーズ（例：ドキュメント整理 ➔ 実装開始）を切り替える際は、必ず作業の手を止め、ユーザーに Git へのコミットを提案すること。
- **ブランチの原則**: 新機能の実装や大幅な修正を開始する前は、必ず `main` から目的別のブランチ（例: `feature/...`）を作成することを提案すること。
- **同期の原則**: 作業を開始する前は、必ず `main` ブランチで `git pull` を行い、最新のリモート状態を反映することを提案すること。
- **判定基準**: `git status` を確認し、未追跡・未コミットのファイルがある状態で次のタスクに進もうとした場合、AI は「一旦ここで記録（コミット）しませんか？」と制止すること。
- **理由**: コンテキスト（文脈）が混ざるのを防ぎ、安全に前の状態に戻れるようにするため。

### 📦 コミット・プルリクエストの規律
- **一機能・一コミット（Atomic Commit）**: コミットは最小単位で「意味のある変更」であること。
- **PRタイトルの標準化**: `[docs]`, `[feat]`, `[fix]` などのプレフィックス（接頭辞）を付け、日本語の概要を添えること。
- **背景と意思決定の明文化**: PR の説明文（Description）には、**`.github/pull_request_template.md`** を必ず活用すること。単なる変更点だけでなく、なぜその設計を選んだのか（例：Wikiではなく分散配置にした理由）や、スキル習得上の目的を含めて丁寧に記述すること。
