# 🏗 現場の視点を取り入れる：標準開発フロー (SDLC)
*SDLC: Software Development Life Cycle（ソフトウェア開発ライフサイクル）*

本ガイドは、将来のチーム開発や大規模プロジェクトへの参画を見据え、個人開発においてもプロの現場で求められる「自走力」と「品質への規律」を身につけるための、**思考のトレーニング・マニュアル** です。

「ただ動くものを作る」段階から、**「なぜそれが必要か、どうやって保守し続けるか」** を論理的に考え、他者（チームメンバーや未来の自分）に説明できる設計力を養うことを目的としています。

---

## ① 要件定義 (Requirements Definition)
*「何を作るか」を、ビジネス上の価値とセットで言語化する。*

### 1-A. 要求分析 (Demand Analysis)
- [ ] **ユーザー要求 (User Needs)**: 解決したい課題や「理想の状態」を具体的に定義したか？
- [ ] **ビジネス目的 (Business Value)**: この開発によって得られる価値（コスト削減、リスク低減等）を明確にし、投資対効果を意識したか？

### 1-B. 要件策定 (Requirement Specification)
- [ ] **入力 (Input)**: システムが受け取るデータとその形式（ログイン情報、URL、ファイル等）を特定したか？
- [ ] **出力 (Output)**: 生成する成果物とその形式（CSV、PDF、保存先フォルダ等）を特定したか？
- [ ] **正常系 (Happy Path)**: 成功時のステップバイステップの処理フローを確定したか？
- [ ] **異常系 (Edge Cases)**: 失敗が予見されるケース（ネット遮断、入力不備等）と、その対処（例外処理、再開フロー等）を決定したか？

### 1-C. デプロイメント戦略（重要：提供形式の決定）
- [ ] **配布形式の選択**: インフラ構成やユーザー環境に合わせ、適切な提供ルートを選択したか？
    - **[Route A: 実行ファイル型]** 不特定多数がダブルクリックで動かすツール (EXE)
    - **[Route B: Web/サーバー型]** 特定のURLで提供されるサービス (SaaS, API)
    - **[Route C: ライブラリ型]** 他のプログラムから呼び出される部品 (Package)
    - **[Route D: 内部スクリプト型]** 特定のインフラで実行される自動化スクリプト

### 1-D. 開発スタイル・規模の決定
- [ ] **開発規模の選択**: 今回の開発は、どのスタイルで進めるべきか？
    - **[Solo]**: 高速プロトタイピング。ドキュメントより「動くコード」重視。
    - **[Team]**: 複数人開発。PR（プルリク）、共通コード規約、レビューを前提。
    - **[Large]**: 大規模・長期運用。自動テスト（CI/CD）の網羅性と疎結合設計を最優先。

### 1-E. 制約条件 (Constraints)
- [ ] **非機能要件**: 「EXE化必須」「日本語ログ必須」など、機能以外のルールを確認したか？
- [ ] **環境要件**: 対応OS、ブラウザの種類やバージョン、推奨スペックなどは決定したか？
- [ ] **Git Checkpoint**: ここまでの要件定義を Git にコミットしたか？

### 1-F. ブランチ戦略 (Branching Strategy)
- [ ] **最新状態の同期 (Pull)**: `main` ブランチに切り替え、リモートリポジトリから最新の変更 (`git pull`) を取り込んだか？
- [ ] **作業ブランチの作成**: `main` から分岐した、目的別のブランチ（`feature/...`, `fix/...`, `docs/...`）を作成したか？

---

## ② 設計 (Design)
*「どう作るか」を、他人が理解できる形式でアウトライン化する。*

### 2-A. 外部設計 (External Design / UI)
- [ ] **ユーザーインターフェース (UI)**: ユーザーがどう触るか、最適な形式を選択したか？
    - **[CLI]**: コマンドライン。引数設計と文字出力が中心。
    - **[GUI]**: 画面操作。レイアウト、ボタン、視線誘導の設計。
    - **[API/NoUI]**: プログラム呼出。データ受渡（JSON等）の厳密な定義。
- [ ] **外部仕様**: 出力されるファイル構成、フォルダのネーミング規則を決めたか？
- [ ] **用語定義 (Terminology)**: 言葉の定義（「成績書」＝「PDF」等）を利用者と揃えたか？

### 2-B. 内部設計 (Internal Design)
- [ ] **アーキテクチャ・部品分割**: 既存部品（`common_utils` 等）の活用、および将来的に再利用可能な「モジュール単位」の分割を検討したか？
- [ ] **データ管理**: プログラム内部でのデータの持ち方（辞書、リスト、クラス等）を決めたか？
- [ ] **例外の伝搬 (Error Handling)**: エラーをどこでキャッチし、ユーザーへどう通知するかを設計したか？
- [ ] **Git Checkpoint**: ここまでの設計内容を Git にコミットしたか？

---

## ③ 実装 (Implementation)
*設計に基づき、規律を持ってコードに落とし込む。*

### 3-A. 環境準備 (Setup)
- [ ] **仮想環境 (venv) の利用**: グローバル環境を汚さぬよう、プロジェクト専用の仮想環境を有効化しているか？
- [ ] **ブランチの確認**: 正しい作業ブランチ（`main`以外）にチェックアウトしているか？
- [ ] **依存ライブラリの管理**: `requirements.txt` 等に必要な外部ツールを追記したか？
- [ ] **構成の確立**: パッケージ階層を整え、`__init__.py` 等を適切に配置したか？

### 3-B. コーディング (Coding Styles)
- [ ] **設計・規模の順守**: 1-D で決めたスタイル、および `GEMINI.md` の掟に従っているか？
- [ ] **静的解析の実施**: Linter (Ruff 等) や型チェック (Pyright/Mypy 等) で、実行前にエラーがないことを確認したか？
- [ ] **可読性向上**: 適切な命名、一貫したログ出力、そして他者が読み解ける丁寧な日本語コメントを実装したか？
- [ ] **Git Checkpoint**: 実装の大きな塊ごとに Git にコミットしたか？

---

## ④ 検証 (Verification)
*要求（①）を満たしているかを、「証拠」とともに確認する。*

### 4-A. テスト (Quality Assurance)
- [ ] **単体テスト**: 主要なロジック（型番の整形、URL生成等）が、期待通り動作するか？
- [ ] **統合/システムテスト**: 全体が連携し、1-B で決めた正常系/異常系の動作を満たすか？

### 4-B. 戦略別・最終検証
- [ ] **UI検証**: GUIなら画面崩れ、CLIなら引数エラーの処理が正しいか？
- [ ] **配布検証**: 1-C で決めたルートに応じた実環境テスト（EXE起動、サーバー動作等）を行ったか？

### 4-C. セルフ・レビュー (Professional Etiquette)
- [ ] **Diff 確認**: `git add` 前に、意図しない `print` や一時的なデバッグコード、不要な改行が混入していないか確認したか？
- [ ] **セキュリティ確認**: `.env` の内容やパスワード、APIキーなどがソースコードにハードコーディング（直書き）されていないか？

---

## ⑤ リリース (Release)
*作ったものを「価値」として他者が使える状態にする。*

### 5-A. ドキュメンテーション
- [ ] **利用マニュアル (README)**: 導入、設定、動作要件を正確に記載したか？
- [ ] **変更履歴 (ChangeLog)**: 変更点、新機能、バグ修正を分かりやすく記述したか？

### 5-B. デプロイメント
- [ ] **最新状態の同期 (Pull before PR)**: PR作成前に、`main` の最新変更を自分のブランチに取り込み、衝突がないか確認したか？
- [ ] **Pull Request の作成**: 作業ブランチをリモートへ `git push` し、GitHub 上で Pull Request を作成したか？
- [ ] **マージの実行**: レビュー（または自己確認）完了後、`main` ブランチへ安全にマージしたか？
- [ ] **配信の実行**: 1-C の戦略に基づき、適切な場所（Releases、本番環境、ライブラリ公開等）へ反映したか？

---

## ⑥ 保守・運用 (Maintenance)
*リリースしたものを「永続的」に機能させるための準備。*

- [ ] **サービス形態の選択**: 運用の深さを定義したか？
    - **[Tool Type]**: 必要に応じて修正。
    - **[Service Type]**: 24/365監視、アラート発報、継続的な改善が必要。
- [ ] **フィードバックループ**: 利用者の声から、新たな「要求（1-A）」を抽出する準備は整ったか？
